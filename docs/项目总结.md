# 基于 MQTT 与阿里云物联网平台的发酵罐远程监控系统设计与实现

## 摘要

本实验围绕工业发酵过程的远程监控需求，设计并实现了一套完整的物联网实验系统。该系统以发酵罐为被控对象，涵盖模拟终端层、云平台层、后端服务层与前端应用层四个核心层级，形成了从边缘设备到用户界面的全链路数据通道。在通信协议的选择上，系统采用 MQTT 作为设备与云平台之间的核心传输协议，利用其发布订阅机制与轻量级报文特性满足物联网场景下低带宽、高并发的通信需求；同时引入 AMQP 作为云平台向后端服务分发消息的队列协议，实现了异构系统间的异步解耦。后端服务基于 Java Spring Boot 构建，通过 Redis 提供高吞吐量的时序数据缓存，并借助 PostgreSQL 完成历史数据的持久化存储；前端采用 React 框架配合 STOMP over WebSocket 实现毫秒级数据推送与可视化呈现。实验结果表明，该系统在架构分层、协议选型与实时性保障等方面均达到了课程设计预期目标，为工业物联网应用的后续研究提供了可复用的技术验证平台。

## 1. 实验背景与目标

发酵工程是现代生物制造领域的核心环节，发酵罐内部的温度、pH 值、溶氧量等参数直接决定了菌体的代谢效率与产物质量。传统的现场监控模式依赖于操作人员定时巡检与手动调节，存在响应滞后、数据离散、人力成本高等固有缺陷。随着物联网技术的成熟，将传感器网络、边缘计算与云端服务深度融合，构建发酵过程的数字化镜像，已成为流程工业智能化转型的重要方向。本实验以发酵罐远程监控为应用场景，旨在通过完整的系统级开发实践，深入理解物联网架构中设备接入、消息路由、数据存储与实时可视化等关键技术的协同机制。

本实验的核心目标可归纳为三个方面。其一，在通信层面验证 MQTT 协议在资源受限设备上的适用性，并探索阿里云物联网平台作为公有云中间件在设备生命周期管理、规则引擎与消息流转中的支撑作用。其二，在数据层面建立高并发写入与实时查询兼顾的存储架构，利用内存数据库应对高频时序数据的低延迟访问需求，同时通过关系型数据库保障历史数据的完整性与可审计性。其三，在应用层面构建响应式的 Web 监控界面，实现设备状态的实时推送、趋势曲线的动态渲染以及远程控制指令的安全下发，从而完成功能闭环。

## 2. 系统总体架构

本系统采用经典的分层架构设计，自上而下可划分为前端应用层、后端服务层、云平台层与边缘终端层。四层之间通过标准化协议进行通信，各层职责单一、耦合度低，具备良好的可扩展性与可替换性。前端应用层面向最终用户，提供基于浏览器的可视化监控与交互入口；后端服务层作为业务逻辑的核心承载体，负责协议适配、数据处理、存储管理与接口暴露；云平台层充当设备与后端之间的通信枢纽，提供设备认证、消息路由与规则转发等基础能力；边缘终端层则通过软件模拟的方式复现物理发酵罐的传感行为与控制响应，作为整个数据链路的起点。

从数据流向的视角审视，系统呈现出典型的"云边端协同"特征。终端模拟器以固定周期采集并上报发酵罐的多维工况参数，数据经 MQTT 协议传输至阿里云物联网平台；平台通过内置的规则引擎将设备消息转投至 AMQP 队列，后端服务以消费者的身份持续拉取消息，完成解析后分别写入 Redis 缓存与 PostgreSQL 数据库，并同步经由 WebSocket 通道推送至前端；前端在接收到实时数据后更新界面状态并绘制趋势曲线。与此同时，操作人员在监控界面下发的控制指令沿 HTTP 路径送达后端，后端调用阿里云 OpenAPI 将指令经 MQTT 反向投递至目标设备，从而完成控制闭环。这一双向数据通路的设计，既保证了监测数据的上行实时性，也确保了控制指令的下行可靠性。

## 3. 技术栈选型与论证

技术栈的选取遵循"场景驱动、成熟优先、适度前瞻"的原则，在每一层级的技术决策背后均有明确的工程权衡。

在边缘终端层，系统采用 TypeScript 作为开发语言，运行于 Node.js 运行时之上。TypeScript 的静态类型系统能够在编译阶段捕获大量潜在错误，对于需要精确建模设备属性与控制逻辑的场景尤为适用。模拟器依赖 `aliyun-iot-device-sdk` 完成与阿里云 IoT 平台的 MQTT 连接、属性上报与命令接收，该 SDK 封装了设备三元组认证、保活心跳、断线重连等底层细节，使开发者得以聚焦于业务逻辑而非协议栈实现。交互层面采用 `inquirer` 库构建命令行界面，支持设备的动态增删、参数查看与手动控制，为实验调试提供了灵活的操作入口。

云平台层选用了阿里云物联网平台作为托管服务。相较于自建 MQTT Broker 与设备注册中心，公有云平台在设备认证、Topic 权限管理、规则引擎与数据流转等方面提供了开箱即用的能力，显著降低了实验环境搭建的复杂度。具体而言，平台支持的基于 HMAC-SHA1 的设备密钥认证机制，确保了终端接入的安全性；其规则引擎允许将满足特定条件的设备消息转发至 AMQP、RocketMQ 或函数计算等多种下游服务，本实验选择 AMQP 作为后端消费端口的协议，正是看中了其与 Spring 生态中 JMS 编程模型的天然亲和性。

后端服务层基于 Spring Boot 4.0.6 构建，该框架在依赖注入、自动配置与面向切面编程等方面的成熟支持，使得后端代码能够以高度模块化的方式组织。数据访问层面采用了双存储策略：Redis 6.x 作为内存数据库，负责缓存最近二十分钟的设备状态时序数据，利用其有序集合数据结构按时间戳排序，实现了滑动窗口式的高效读写；PostgreSQL 15 作为关系型数据库，承担历史数据的持久化职责，并通过 JPA/Hibernate 完成对象关系映射，简化了数据访问层的编码工作。此外，Spring 框架内置的 `@Scheduled` 定时任务机制被用于周期性下采样操作，将 Redis 中的最新状态按固定周期归档至 PostgreSQL，从而在实时性与存储成本之间取得了平衡。实时推送方面，系统选用 Spring 的 STOMP over WebSocket 实现方案，相较于原生 WebSocket，STOMP 协议在消息语义、订阅管理与错误处理等方面更为完备，配合前端的 `@stomp/stompjs` 库，能够建立稳定的全双工通信通道。

前端应用层基于 React 19 构建，采用 Vite 作为构建工具以获得极速的冷启动与模块热更新体验。样式系统选用 Tailwind CSS，通过原子化类名的组合方式快速构建响应式界面，避免了传统 CSS 方案中样式冲突与命名空间污染的问题。数据可视化环节引入 ECharts 6.x，利用其折线图与面积图组件渲染温度、pH 值、溶氧量与泡沫液位等关键参数的实时趋势。需要指出的是，前端并未引入额外的状态管理库，而是充分利用 React 内置的 `useState` 与 `useCallback` Hook 完成局部状态的管理与事件回调的优化，这在当前单页面、低交互复杂度的场景下已完全足够，避免了引入 Redux 等重型方案所带来的样板代码开销。

## 4. 各子系统详细设计

### 4.1 终端模拟层：发酵罐网关模拟器

终端模拟层的核心职责是以软件方式复现物联网网关与下游发酵罐设备的行为。在真实工业场景中，网关负责汇聚多个传感节点的数据并通过单一 MQTT 连接上报云端；本实验将网关与终端设备的功能合而为一，以简化部署流程。模拟器内部维护一个 `Fermenter` 实例集合，每个实例代表一台独立的发酵罐终端，拥有唯一的设备三元组（`productKey`、`deviceName`、`deviceSecret`），通过阿里云设备 SDK 建立持久化的 MQTT 连接。

在属性建模方面，每台发酵罐维护十维状态空间，涵盖四类传感读数与五类执行器输出，外加一个控制模式标志位。传感读数包括温度、pH 值、溶氧量与泡沫液位，其数值在每次更新周期内引入随机扰动以模拟真实环境的测量噪声与过程波动。执行器输出包括加酸泵、加碱泵、冷却阀、加热阀与搅拌电机的开度百分比，这些参数既可通过本地控制策略自动生成，也可接受来自云端的远程设定值。控制模式标志位划分为本地自动（`LOCAL`）与远程控制（`REMOTE`）两种状态：处于本地模式时，系统内置的模糊控制器根据当前温度与 pH 值实时计算执行器输出，以维持发酵环境的稳态；切换至远程模式后，执行器输出完全由后端通过 OpenAPI 下发的属性设置指令决定，这一设计为后续接入真实 PID 或 MPC 控制算法预留了扩展接口。

模糊控制器的实现采用了基于规则表的简化策略。对于 pH 控制回路，控制器将测量值划分为偏碱、过碱、偏酸与过酸四个区间，分别对应加酸泵与加碱泵的不同增益档位；对于温度控制回路，则将测量值划分为偏热、过热、偏冷与过冷四个区间，分别调节冷却阀与加热阀的开度。尽管该模糊控制器的论域划分与规则数量较为精简，但其设计思想与实际工业模糊控制系统保持一致，即通过语言变量与模糊规则将操作人员的经验知识形式化为可自动执行的策略，适用于难以建立精确数学模型的复杂生物反应过程。

### 4.2 云平台层：阿里云物联网平台集成

阿里云物联网平台在本系统中扮演通信中间件的角色，向上对接终端设备的 MQTT 连接，向下通过 AMQP 协议将消息投递至后端服务。设备接入采用一机一密的认证模式，每个模拟终端在创建时由平台分配唯一的 `deviceSecret`，终端 SDK 在建立 MQTT 连接前使用该密钥对连接请求进行 HMAC-SHA1 签名，平台侧完成签名校验后才会授予发布与订阅权限。这种基于非对称签名机制的认证方式，相较于固定密码或 Token 方案，能够有效防止重放攻击与凭证泄露风险。

在消息流转路径上，终端通过 MQTT 的 `POST` 方法向预设 Topic 上报属性数据，平台的规则引擎监听该 Topic 并将消息内容转发至绑定的 AMQP 队列。后端服务以长连接方式消费该队列，利用 Apache Qpid JMS 客户端库与阿里云 AMQP 服务端建立持久会话，并在连接断开后自动重连。AMQP 协议相较于 MQTT 在消息语义上更为丰富，支持消息持久化、投递确认与流量控制等高级特性，这使其更适合作为服务端之间的异步通信协议。后端在消费消息时，依据消息体中是否包含 `items` 字段或 `status` 字段，区分属性上报消息与设备连接事件消息，并分别交由不同的业务处理路径。

控制指令的下发路径与数据上报路径呈镜像关系。后端在接收到前端的控制请求后，调用阿里云物联网平台的 OpenAPI 接口 `SetDeviceProperty`，通过标准的 HTTP GET 请求将待设置的属性键值对传递至平台；平台随后将属性设置指令通过 MQTT 下发至目标设备。OpenAPI 的调用遵循阿里云通用的签名机制，所有请求参数按字典序排列后拼接为待签名字符串，再使用 AccessKey Secret 进行 HMAC-SHA1 签名，最终生成的一次性 URL 既保证了请求的不可伪造性，也避免了敏感凭证在传输过程中的暴露。

### 4.3 后端服务层：业务逻辑与数据管理

后端服务层是整个系统的技术枢纽，承担了协议适配、业务处理、数据存储与实时推送四项核心职能。在代码组织结构上，后端遵循 Spring 框架推崇的分层架构：控制器层（Controller）负责处理 HTTP 与 WebSocket 请求并封装响应；服务层（Service）承载核心业务逻辑；数据访问层（Repository）屏蔽底层存储差异；实体层（Entity）与数据传输对象层（DTO）则完成了领域模型在不同上下文中的精确映射。

消息处理是后端的核心入口。`AmqpMessageListener` 实现了 JMS 的 `MessageListener` 接口，在 `onMessage` 方法中解析字节流形式的 JSON 报文。对于属性上报消息，系统提取 `deviceName` 与 `items` 字段，将各属性的当前值封装为 `FermenterStatusDto`，随后同时执行三项操作：其一，将 DTO 序列化后写入 Redis 有序集合，以当前时间戳作为分数，利用 Redis 的原子性范围删除操作自动淘汰二十分钟前的过期数据；其二，通过 `SimpMessagingTemplate` 将 DTO 广播至 `/topic/fermenter-status` 主题，所有订阅该主题的 WebSocket 客户端将即刻收到更新；其三，返回消费确认，确保消息不会重复投递。对于设备连接事件消息，系统提取 `status`、`iotId` 与 `lastTime` 等字段，将上线或离线事件持久化至 PostgreSQL，并同步更新 Redis 中的设备在线状态缓存，以便前端在渲染设备列表时能够即时反映各终端的连通性。

数据存储策略的设计体现了对访问模式的深刻洞察。Redis 中的有序集合为每个设备维护一条时间序列，支持按分数范围快速检索任意时间窗口内的状态快照，这恰好满足了前端对最近二十分钟趋势数据的查询需求。然而，内存存储的成本远高于磁盘存储，且 Redis 的持久化机制在极端故障场景下仍可能存在秒级数据丢失的风险。为此，系统引入了定时下采样机制：借助 Spring 的 `@Scheduled(cron = "0 30 * * * *")` 注解，后端在每个整点三十分钟时触发一次归档任务，将 Redis 中各设备的最新状态记录转存至 PostgreSQL 的 `fermenter_status` 表中。该表以自增主键为标识，以 `timestamp` 字段为时序索引，支持按设备名称分页查询历史记录。通过这种"热数据驻留内存、冷数据落盘归档"的分层存储策略，系统在不牺牲实时查询性能的前提下，实现了数据全生命周期的经济化管理。

设备连接管理是后端的另一项重要职责。`FermenterConnectionService` 负责解析设备上下线事件并维护在线状态视图。当终端成功建立 MQTT 连接时，阿里云平台会生成包含 `status: online` 的 AMQP 消息；当连接断开或心跳超时时，则生成 `status: offline` 消息。后端在接收到此类事件后，将事件详情写入 `fermenter_connection_event` 表，形成可供审计的设备连接日志，同时更新 Redis 哈希结构中的在线标志位、最近活跃时间与 `iotId` 映射。`iotId` 是阿里云平台为每个设备实例分配的全局唯一标识，后端在接收到前端的控制指令后，需要首先通过 `deviceName` 查询对应的 `iotId`，再以此调用 OpenAPI，这一间接映射的设计保证了设备标识体系与云平台内部机制的一致性。

### 4.4 前端应用层：可视化监控与交互

前端应用层作为用户与系统交互的唯一界面，其设计目标是在有限的屏幕空间内以直观、实时、可操控的方式呈现发酵过程的全貌。应用采用单页面架构，整体布局由左侧设备导航栏与右侧主内容区构成。导航栏以列表形式展示当前所有已注册设备的名称与在线状态，用户点击设备名称后，主内容区切换至该设备的详情视图。

设备详情视图从上至下依次呈现四组信息模块。最上方为关键参数卡片，以四宫格布局展示温度、pH 值、溶氧量与泡沫液位的最新读数，每个卡片通过颜色编码与动态数值呈现当前工况的健康程度。中间区域为控制面板，提供本地自动与远程控制两种模式的切换按钮，以及加酸、加碱、冷却、加热与搅拌五个执行器的滑块调节器。当设备处于本地自动模式时，滑块被禁用，系统提示当前由网关模糊控制器接管；切换至远程控制模式后，用户可通过拖动滑块实时调节执行器开度，每次释放滑块后前端即向后端发送包含目标设备名称与属性键值对的 POST 请求，后端随后通过 OpenAPI 将指令下发至设备。下方区域为趋势图表区，采用二乘二网格布局展示温度、pH 值、溶氧量与泡沫液位在最近二十分钟内的变化曲线，图表基于 ECharts 的面积图组件实现，X 轴为时间刻度，Y 轴为参数数值，曲线下方填充渐变色以增强视觉层次感。

实时数据的获取采用"拉取初始化、推送增量"的混合策略。当用户首次选中某设备时，前端通过 HTTP GET 请求向后端拉取该设备最近二十分钟的历史数据，用于初始化趋势图表与参数卡片；与此同时，前端通过 STOMP 协议与后端的 WebSocket 端点建立持久连接，并订阅 `/topic/fermenter-status` 主题。此后，每当后端接收到来自 AMQP 队列的新消息，便会通过 `SimpMessagingTemplate` 将数据广播至该主题，前端在 `onMessage` 回调中解析 JSON 载荷，更新状态变量并触发 React 的重新渲染机制。为保障 WebSocket 连接的稳定性，前端配置了五秒重连延迟与双向心跳机制，一旦连接因网络抖动而断开，STOMP 客户端将在后台自动尝试恢复连接，用户无需手动刷新页面。

## 5. 关键技术分析

### 5.1 多协议协同机制

本系统在一个完整的业务流程中同时涉及 MQTT、AMQP、HTTP 与 WebSocket 四种协议，这一多协议架构并非简单的技术堆砌，而是对各协议特性与适用场景的精准匹配。MQTT 以其极低的报文开销与发布订阅语义，成为设备侧网络库的首选；AMQP 以其可靠的消息队列语义与丰富的企业级特性，成为服务端之间的异步通信桥梁；HTTP 以其无状态的请求响应模型与广泛的中间件支持，成为前端与后端之间控制指令交互的标准选择；WebSocket 则以全双工通信能力弥补了 HTTP 在服务端推送场景下的先天不足。四种协议各司其职、无缝衔接，共同构成了从边缘到云、从云到端的数据通路。

### 5.2 时序数据的内存缓存策略

发酵过程的实时监控对数据查询延迟提出了苛刻要求，传统关系型数据库在面对高频写入与范围查询的混合负载时往往难以兼顾。本系统选择 Redis 有序集合作为时序数据的内存缓存层，其设计精髓在于利用分数排序特性天然支持按时间范围检索。每条设备状态记录被序列化为 JSON 字符串后存入有序集合，以毫秒级时间戳作为分数；查询最近二十分钟数据时，只需计算当前时间与二十分钟前的时间戳边界，通过 `ZRANGEBYSCORE` 命令即可在常数级时间内返回结果。同时，利用有序集合的 `ZREMRANGEBYSCORE` 命令在每次写入时同步淘汰过期数据，确保了内存占用量的有界性。这一滑动窗口缓存策略在日志分析、物联网监控等时序密集型场景中具有广泛的适用性。

### 5.3 控制闭环的安全性考量

远程控制指令的下发涉及跨网络、跨系统的安全边界。本系统在控制通路上设置了多重保障机制：其一，阿里云 IoT 平台的设备认证体系确保了只有持有合法密钥的终端才能接入并接收指令；其二，后端在调用 OpenAPI 前会校验目标设备的在线状态与 `iotId` 映射关系的有效性，防止向不存在的设备发送无效指令；其三，HTTPS 协议为 OpenAPI 调用提供了传输层加密，避免了控制指令在公网传输过程中被窃听或篡改；其四，前端在发送控制请求时仅传递 `deviceName` 而非 `iotId`，将平台内部标识的映射逻辑隐藏在后端，降低了接口暴露面。这些措施共同构成了纵深防御的安全架构，为后续向真实工业场景迁移奠定了信任基础。

### 5.4 本地智能与云端协同的控制架构

系统在控制策略的设计上采用了"边缘自治、云端可调"的混合架构。本地自动模式下，网关内置的模糊控制器依据实时传感读数独立计算执行器输出，无需依赖网络连接即可维持发酵环境的稳定运行，这一设计在工业现场网络不可靠的场景下尤为重要。远程控制模式下，操作人员在监控界面发出的调节指令通过云端直达设备，实现了对局部控制回路的实时干预。两种模式之间的切换由 `controlMode` 属性位控制，设备在接收到模式切换指令后即时变更控制权的归属，整个过程对上层业务透明。这种将实时控制闭环下沉至边缘、将监控与优化决策上浮至云端的架构思想，与当前工业界倡导的"边缘计算 + 云计算"协同范式高度契合。

## 6. 总结

本实验围绕发酵罐远程监控这一典型工业物联网场景，设计并实现了一套涵盖终端模拟、云平台接入、后端服务与前端可视化的完整系统。在技术实现层面，系统验证了 MQTT 协议在设备接入场景中的高效性，探索了 AMQP 队列在服务端消息流转中的可靠性，构建了基于 Redis 有序集合的滑动窗口时序缓存机制，并通过 STOMP over WebSocket 实现了前后端的实时数据同步。在架构设计层面，系统采用分层解耦的组织方式，各层级之间通过标准化协议交互，既保证了当前功能的完整性，也为后续模块的独立演进预留了空间。在控制策略层面，实验将模糊控制算法嵌入边缘模拟器，形成了本地自治与远程干预的双模式控制架构，体现了边缘计算在工业控制领域的应用价值。总体而言，本实验不仅达成了课程设计的预设目标，更为深入理解物联网系统的全栈开发流程提供了宝贵的工程实践经验。
